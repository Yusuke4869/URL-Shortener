# 開発ガイドライン

開発・運用に関するガイドライン集です。

## はじめに

新しいドキュメント作成や既存ドキュメントの編集を行う前に、必ず[ドキュメント作成](./documentation.md)をお読みください。
統一された形式でドキュメントを作成するための重要なルールが記載されています。

## プロジェクト構造

**ディレクトリ命名規則**: 全てのディレクトリ名は単数形を使用します。

```
.
├── docs/                          # ドキュメント
│   ├── README.md                  # このファイル
│   ├── architecture.md            # アーキテクチャ
│   ├── coding-standards.md        # コーディング規約
│   ├── api-design.md             # API設計
│   ├── refactoring.md            # リファクタリング指針
│   └── documentation.md           # ドキュメント作成
├── src/
│   ├── controller/               # HTTP リクエスト処理層
│   ├── domain/                   # ドメインエンティティ
│   ├── infrastructure/           # 外部システム接続層
│   ├── middleware/               # 横断的関心事層
│   ├── repository/               # データアクセス層
│   ├── route/                    # ルート定義層
│   ├── service/                  # ビジネスロジック・ドメインサービス層
│   ├── usecase/                  # アプリケーションロジック層
│   ├── util/                     # ユーティリティ層
│   └── main.ts                   # エントリーポイント
├── public/                       # 静的ファイル
├── deno.json                     # Deno 設定
├── LICENSE                       # ライセンス
└── README.md                     # プロジェクト説明
```

## 技術スタック

- **ランタイム**: Deno
- **言語**: TypeScript
- **Web フレームワーク**: Hono
- **データベース**: Deno KV
- **バリデーション**: Zod
- **テスト**: Deno 標準テストフレームワーク
- **デプロイ**: Deno Deploy

---

**重要**:
新機能開発や重要な変更を行う前に、必ず該当するドキュメントを確認してください。
これにより、プロジェクトの一貫性と品質が保たれます。

## 開発ガイドライン品質チェックリスト

新規ドキュメント作成や既存ドキュメント更新時の確認事項：

### 構造と形式

- [ ] 適切な見出し構造（H1-H6 の階層）
- [ ] 見出しの前後に空行
- [ ] コードブロックの前後に空行
- [ ] 箇条書きの前後に空行
- [ ] 英単語の前後に半角空白
- [ ] 1 文ごとに改行
- [ ] 技術用語の統一

### 内容の品質

- [ ] 目的と概要が明確
- [ ] 実在するファイル・コード例のみ記載
- [ ] 内部リンクが正確（相対パスを使用）
- [ ] 外部リンクが有効
- [ ] 記述スタイルの統一

### プロジェクト固有

- [ ] Deno と TypeScript の使用に適した内容
- [ ] クリーンアーキテクチャの原則に準拠
- [ ] 実際のディレクトリ構造と一致
- [ ] 実際の API 仕様と一致
- [ ] セキュリティベストプラクティスの適用
